LionClaw

Local AI CLI

Why What You Run Core vs Skills Links

Local AI with a real boundary.

LionClaw gives you one command for everyday local AI work while keeping policy, audit, state, and runtime control inside a small trusted core. Channels and automation stay outside that core until you choose to add them. 

./target/debug/lionclaw onboard
./target/debug/lionclaw runtime add codex --kind codex --bin codex
./target/debug/lionclaw run codex

Start with the CLI. Add channels or background automation later as skills. The trusted core stays small either way.

Why LionClaw

Keep the trusted core small as the system grows.

Local AI gets more useful as it gains memory, channels, automation, and runtime control. Those capabilities do not all belong inside the part of the system you have to trust.

LionClaw keeps policy, audit, state, and runtime control in the core. Channels and future capabilities stay outside it as skills, so the everyday path stays simple while the boundary stays clear.

One command for daily work. Optional skills when you need more.

Start Here

Start with one command.

Build the binaries, onboard once, add a runtime, and use lionclaw run for normal interactive work. When you come back, you can continue the latest session instead of starting over. 

cargo build --bins
./target/debug/lionclaw onboard
./target/debug/lionclaw runtime add codex --kind codex --bin codex
./target/debug/lionclaw run codex
./target/debug/lionclaw run --continue-last-session codex

Need channels or background automation later? Add them as skills. The CLI path stays the default.

What Stays Trusted

The core stays small on purpose.

Trusted core

  • policy and audit
  • durable state for sessions and runtime control
  • one local entrypoint for interactive AI work

Optional skills

  • channels and automation you add later
  • future capabilities without inflating the core
  • background work only when explicitly turned on
