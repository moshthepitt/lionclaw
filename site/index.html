Local AI. Small trusted core.

One CLI you actually use. One core you can reason about. Everything beyond that core installed as a skill. 

./target/debug/lionclaw onboard
./target/debug/lionclaw runtime add codex --kind codex --bin codex
./target/debug/lionclaw run codex

The everyday path is simple: run LionClaw locally, keep policy and state in the trusted core, and turn on channels or background work only when you ask.

Why LionClaw

Most agent tools make the wrong tradeoff.

They either grow into sprawling always-on systems you do not understand, or they stay thin and leave the model sitting too close to your machine.

LionClaw takes a harder line. Keep the trusted core small. Put policy, audit, state, and runtime control there. Push everything else outward.

Channels are skills. Future capabilities are skills. The core is small on purpose.

What You Run

One obvious command path.

LionClaw is built around a direct local CLI path. You build the binaries, onboard once, add a runtime, and use lionclaw run for normal interactive work. 

cargo build --bins
./target/debug/lionclaw onboard
./target/debug/lionclaw runtime add codex --kind codex --bin codex
./target/debug/lionclaw run codex
./target/debug/lionclaw run --continue-last-session codex

When you want a real channel instead of the direct CLI path, install one as a skill. For a local end-to-end test without Telegram: 

./scripts/bootstrap-terminal-test.sh /tmp/lionclaw-terminal-e2e

Core vs Skills

The boundary is the product.

Trusted core

  • policy and audit
  • durable state for sessions, channels, and runtime control
  • one local entrypoint for interactive AI work

Installed skills

  • channels such as terminal or Telegram
  • future capabilities added without inflating the core
  • background operation only when explicitly turned on
